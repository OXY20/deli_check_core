前端安全

上传内容进入 innerHTML，存在 XSS 注入面

高风险
位置
web/static/results.html:417-445、web/static/results.html:485-528、web/static/results.html:583-612
问题
Excel 中的员工姓名、工号、部门、日期、时间、地点等字段被拼接到模板字符串，再写入 innerHTML，没有统一 HTML 转义。
影响
恶意 Excel 单元格可注入 HTML/JS。本地工具也有风险，因为脚本可读取同源 sessionStorage 中的考勤数据，并调用同源接口。
建议
所有来自 Excel、接口、sessionStorage 的字段统一转义，或改为使用 textContent 创建 DOM，避免直接拼接 innerHTML。
反馈：用户意见：待确认是否优先修复。追加建议：这是当前最值得优先处理的问题。实施状态：[ ] 待实施

结果数据保存在 sessionStorage，受 XSS 影响面放大

中风险
位置
web/static/index.html:554、web/static/index.html:618、web/static/results.html:369
问题
完整考勤结果保存在浏览器会话存储中，页面刷新和结果页展示依赖该数据。
影响
一旦出现 XSS，脚本可以直接读取完整员工考勤记录。
建议
短期先修 XSS；中长期可考虑服务端临时会话 ID，前端不保存完整敏感数据。
反馈：用户意见：待确认是否需要服务端会话化。追加建议：本地工具可先保留当前机制，但必须先封住 XSS。实施状态：[ ] 待确认

接口与本地 Web 安全

上传接口缺少整体请求体限制和文件数量限制

高风险
位置
web/server.go:357-465
问题
ParseMultipartForm(128 << 20) 不是完整的请求体防护，没有额外限制上传文件数量、单文件大小和总请求大小。
影响
恶意请求可提交大量文件或大文件，造成内存、磁盘、CPU 消耗过高。
建议
使用 http.MaxBytesReader 限制总请求体，并限制文件数量、单文件大小和字段长度。
反馈：用户意见：待确认具体业务上限。追加建议：根据考勤表实际大小设置较小阈值，不建议保留 128MB 默认值。实施状态：[ ] 待实施

导出接口完全信任前端提交的 JSON

高风险
位置
web/server.go:146-245、web/server.go:247-331
问题
/api/export-meal 和 /api/export-personnel 直接接收前端 records 并生成 Excel，没有限制 JSON 体大小、记录条数和字段长度。
影响
可被构造超大请求导致内存膨胀或生成超大 Excel，也可能导出并非真实解析来源的数据。
建议
增加 JSON 请求体限制、记录条数限制、字段长度限制；长期考虑服务端保存解析结果，导出时按临时 ID 获取。
反馈：用户意见：待确认是否需要服务端会话化。追加建议：短期先做大小限制，长期再考虑服务端状态管理。实施状态：[ ] 待实施

本地 POST 接口缺少 Origin/Referer 校验

中风险
位置
web/server.go:40-46、web/server.go:146-465
问题
虽然服务监听 127.0.0.1，但本机浏览器中的恶意网页仍可尝试向本地端口提交请求。
影响
攻击者不一定能读取响应，但可能触发上传、解析、导出等资源消耗操作。
建议
对非 GET 接口校验 Origin 或 Referer，仅允许当前本地服务来源；或增加一次性 token。
反馈：用户意见：待确认是否接受轻量 Origin 校验。追加建议：本地工具优先做 Origin/Referer 校验，改动小、收益高。实施状态：[ ] 待实施

文件上传与 Excel 解析

上传文件名缺少清洗

中风险
位置
web/server.go:418-423
问题
保存文件名由原始上传文件名派生，只使用 filepath.Base 降低路径穿越风险，但没有处理超长名称、控制字符、Windows 保留名和特殊字符。
影响

建议
对基础文件名做白名单清洗并限制长度，必要时使用 upload_001.xlsx 之类的安全文件名。
反馈：用户意见：待确认是否保留原文件名可读性。追加建议：保留前 40 个安全字符即可，兼顾可读性和安全。实施状态：[ ] 待实施

.xls 解析存在潜在空指针风险

高风险
位置
tools/excel.go:48-53
问题
sheet.Row(i) 可能返回 nil，随后直接调用 row.Col(j) 可能 panic。
影响

补齐 row == nil 判断，空行保留空矩阵即可。
反馈：用户意见：待确认是否优先修。追加建议：这是小改动高收益问题，建议与安全修复一起处理。实施状态：[ ] 待实施

中风险
位置
tools/xlsx.go:11-24
问题

反馈：用户意见：待确认可接受的最大 Excel 文件规模。追加建议：先做上传大小限制，再做解析行列限制。实施状态：[ ] 待实施

数据隐私与落盘

中风险
位置
web/server.go:370-438、core/compose.go:157-168
问题

反馈：用户意见：待确认是否需要保留历史上传。追加建议：默认删除上传源文件，调试模式才保留。实施状态：[ ] 待实施

建议检查方向

输入安全

检查上传文件大小、文件数量、文件名、扩展名、MIME/文件头、Excel 解压炸弹、异常格式、空行、超大行列数。

前端安全

检查所有 innerHTML、模板字符串、sessionStorage 数据渲染、下载文件名解析、XSS、DOM 注入。

本地 Web 安全

检查监听地址、Origin/Referer、CSRF、CORS、接口方法限制、请求体大小限制、错误信息泄露。

数据隐私

检查上传源文件、输出 JSON、日志、临时文件、浏览器 sessionStorage 是否残留员工敏感信息。

业务正确性

检查员工数统计口径、无打卡员工处理、多表合并工号规则、同名员工冲突、日期范围跨月、时间格式异常。

稳定性

检查空指针、panic、第三方库解析异常、文件句柄关闭、并发请求、端口探测、自动打开浏览器失败处理。

依赖供应链

检查 go.mod Go 版本、github.com/extrame/xls 维护状态、excelize 安全公告、go.sum 一致性。

测试发布

检查是否有 *_test.go、构建脚本、版本号一致性、README 与实际行为一致性、样例数据和二进制产物是否误提交。

建议修复优先级

  1. 第一优先级：修复 results.html 的 XSS 渲染问题，给所有接口增加请求体、记录数、字段长度限制。
  2. 第二优先级：给 POST 接口加 Origin/Referer 校验，上传文件名清洗，处理完成后清理上传文件。
  3. 第三优先级：修复 .xls 空行 nil panic，明确员工统计口径，修正文档与实际行为不一致。
  4. 第四优先级：补核心解析单测和 Web handler 边界测试，评估 .xls 依赖替换或隔离策略。
反馈：用户意见：待进一步确认修复范围。追加建议：建议先从高风险且改动小的问题开始，避免一次性重构影响当前功能。实施状态：[ ] 待确认
